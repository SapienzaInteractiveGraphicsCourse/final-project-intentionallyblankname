\section{Project Overview}
% =============================================================================

\textit{Mecha-Basketball 3D} is a real-time basketball match between procedurally generated robots, rendered on a GLTF court with Three.js. The player picks a robot class, each with its own stats and special move, then controls movement, dribbling, aiming and shooting directly; nothing is a scripted replay.

No external model is used for the robots: each class is built in Three.js from primitives, organized as hierarchical \code{THREE.Group} trees and animated procedurally. Only two assets are external: the court and the ball (both GLTF, Section~\ref{sec:env}). All three classes are fully playable by the human player and by the enemy AI, and are swappable from the Main Menu with no page reload. Every class shares one stat struct, \verb|{ speed, shooting, steal, block }|, driving both gameplay and the comparison bars on the selection screen.

\label{sec:structure}
% =============================================================================

The codebase is organized by domain under \code{src/}. \code{main.js} is the only top-level file: it owns the scene, the render loop and every piece of global state, and wires the other modules together (Section~\ref{sec:arch}).

% =============================================================================
\section{Development Environment, Libraries and Assets}
\label{sec:env}
% =============================================================================

Built with \textbf{Three.js} (\code{\^{}0.178.0}) on \textbf{Vite} (\code{\^{}6.0.0}), plain JavaScript ES modules, no TypeScript or framework. Three.js was chosen over raw WebGL to spend effort on modelling and animation rather than shader/buffer boilerplate. \code{vite.config.js} sets \verb|base: './'| so the bundle works from a GitHub Pages subpath.

The renderer targets soft, physically plausible shadows: \code{PCFSoftShadowMap} (VSM was tried and is too slow at $4096^2$), ACES filmic tone mapping, sRGB output.

Beyond the Three.js core, only official addons are used: \code{GLTFLoader}, \code{PointerLockControls}, the \code{EffectComposer} post-processing stack (Section~\ref{sec:postfx}) and the procedural \code{Sky} skydome. No physics engine and no animation library (tween.js, \code{AnimationMixer}) are integrated: collision response, the dribble state machine and every joint animation are hand-written inside the render loop.

Two external, pre-made assets:
\begin{itemize}[nosep]
    \item \textbf{Court} (Sketchfab, modified): a custom \code{GLTFLoader} plugin re-implements \code{KHR\_materials\_pbrSpecularGlossiness} (removed from Three.js r152+); some material colors were hand-edited in the raw \code{scene.gltf}; line z-fighting fixed with \code{polygonOffset}.
    \item \textbf{Basketball}: a dedicated GLTF with color, normal and metalness/roughness maps.
\end{itemize}

Everything else, all three robots (geometry, materials, textures), the skybox and every sound effect, is generated procedurally at runtime.

\section{Architecture and Recurring Patterns}
\label{sec:arch}
% =============================================================================

\code{main.js} owns all shared mutable state (\code{manipulator}, \code{basketball}, \code{cameraState}, \ldots) and is the only file allowed to import from every other module; every gameplay module receives what it needs through a single \emph{context object} instead of importing from \code{main.js}, which would create circular dependencies:

\begin{verbatim}
main.js
  const gameContext = { camera, scene, sfx, controls, cameraState, ... }

  initBallPossession({ ...gameContext, dribbleState, shootingState })
  initShootingSystem({ ...gameContext, getManipulator, getShotDirection })
  initCombatMoves  ({ ...gameContext, stealState, blockState })
  initMainMenu     ({ ...gameContext, ... })

  each init returns { updateX, triggerY, ... }, called back from animate()
\end{verbatim}

A live reference such as the active robot cannot be a plain captured value: it changes at runtime when the player swaps class, and \code{basketball} itself is \code{null} until its GLTF finishes loading asynchronously. Both are passed as zero-argument \emph{getter functions} (\code{getManipulator()}, \code{getBasketball()}) instead, so every consumer always reads the current value.

The render loop drives two independent dispatch chains, one per side (player, enemy):

\begin{verbatim}
animate()
  updateTimeOfDayTransition()
  updateShootAnimation()            windup / release / recover
  updateBallDispatch(player)
    pickup active?      -> updatePickup()
    shot released?       -> updateShotFlight() + checkForPickup()
    no ball?              -> checkForPickup()
    else                   -> updateHandling()  or  updateDribble()  (fixed-step)
  [1v1] updateEnemyAI() -> triggerShoot / triggerSteal / triggerBlock
  [1v1] updateBallDispatch(enemy)
  updateSteal() / updateBlock()      both robots
  updateBallSpin()
  updateTrajectoryPreview()
  composer.render()
\end{verbatim}

Four patterns recur across the codebase:

\begin{itemize}[nosep]
    \item \textbf{Finite state machines everywhere}: robot state (\code{DRIBBLE}/\code{HANDLING}/\code{NO\_BALL}), ball state (\code{HANDLED}/\code{FREE\_SHOT}/\code{FREE}), the dribble (\code{push}/\code{drop}/\code{rise}), the shot (\code{windup}/\code{release}/\code{recover}) and combat moves (\code{reach}/\code{resolve}) are all plain objects with a \code{phase} string and a \code{phaseT} timer, advanced by hand each frame. Enums are frozen plain objects (\code{Object.freeze}), JavaScript has no native enum.
    \item \textbf{Fixed-timestep accumulator}: the dribble integrates at a constant 120\,Hz and the shot flight at 240\,Hz, decoupled from the variable render \code{delta}, via an accumulator (Section~\ref{sec:animation}).
    \item \textbf{Scratch objects}: every hot-path function (collision resolution, trajectory preview, ball spin) reuses \code{THREE.Vector3}/\code{Box3} instances allocated once, instead of allocating per call; the trajectory preview alone can call the collision resolver up to $2400\times$ per frame while aiming.
    \item \textbf{Composition over inheritance for robots}: \code{RobotBase} is a class that wraps a plain factory function (\code{Object.assign(this, factory())}), keeping procedural geometry construction (never touched once stable) separate from gameplay state (stats, FSM, special moves).
\end{itemize}

\section{Hierarchical Models}
\label{sec:models}
% =============================================================================

All three robot classes carry the same 3-revolute-joint arm above (or, for the Drone, hanging below) their locomotion base:
\begin{center}
\small
$\text{base (yaw)} \rightarrow \text{link1} \rightarrow \text{elbow (pitch)} \rightarrow \text{link2 (tapered)} \rightarrow \text{wrist (pitch)} \rightarrow \text{end effector + paddle}$
\end{center}
each joint a nested \code{THREE.Group}, so rotating a parent (e.g.\ the elbow) carries every child with it: the standard scene-graph technique for an articulated model. \code{link2} uses a tapered cross-section from a 4-sided \code{CylinderGeometry} (\code{radialSegments}~$=4$ degenerates the cylinder into a square prism; different top/bottom radii turn it into a truncated pyramid). The ball's resting point during handling is the convergence point of the two paddle plates' normals, at distance $d / \cos(\alpha/2)$ from the wrist, not the paddle's flat surface center, which visibly interpenetrated the ball.

Per-class hierarchy below the arm:
\begin{itemize}[nosep]
    \item \textbf{Mobile Manipulator}: 4 wheels (tori) under a disc chassis; the wheel group's scale derives from the disc radius, so resizing the chassis keeps the wheels attached.
    \item \textbf{Legged Manipulator}: 4 two-joint legs (hip + knee) at the wheel offsets, each oriented radially outward; 25\% larger overall. The flat foot pivots at its rear edge, the same off-center-pivot trick as the paddle.
    \item \textbf{Drone}: a body with four continuously spinning rotor arms (alternating spin direction between neighbors); the arm hangs upside down through a fixed flip group (\code{rotation.x} $= \pi$) between the yaw base and the first link.
\end{itemize}

\code{RobotBase} wraps each procedural factory, owning the shared \code{RobotState} FSM, the stat-derived \code{speed} getter, and empty special-move hooks each subclass overrides.

\section{Materials, Lighting and Textures}
\label{sec:materials}
% =============================================================================

\subsection{Lights}

1 \code{HemisphereLight} (warm ambient, approximating the original Sketchfab IBL), 1 \code{DirectionalLight} ($4096^2$ shadow map, frustum sized to cover the court), 4 \code{PointLight} (lamp posts) and 2 \code{SpotLight} (hoop fixtures, lit only at Sunset/Night; toggled via intensity, never \code{.visible}, to avoid a shadow-map allocation stutter the first time a light becomes visible). Colors, intensities and sun position follow a \code{TimeOfDay} preset table (Sunrise/Day/Sunset/Night), cross-fading over 2.5\,s instead of snapping.

\paragraph{Scale.} The court uses centimeter-scale units spanning thousands of units, so every physically-motivated Three.js default needed recalibration: the shadow frustum spans $\pm 2500$ units, giving low texel density at $4096^2$ (fixed with \code{shadow.bias}/\code{normalBias}); point-light intensity is candela with inverse-square decay, so lamp-to-ground values around $2 \times 10^5$ were needed at $\sim$250 units; each lamp sits at the center of its own glass globe, self-shadowing unless \code{shadow.camera.near} exceeds the globe radius.

\subsection{Post-Processing}
\label{sec:postfx}

\code{EffectComposer} pipeline: \code{RenderPass} $\rightarrow$ \code{SSAOPass} $\rightarrow$ \code{OutputPass} $\rightarrow$ \code{SMAAPass}. \code{SSAOPass.minDistance}/\code{maxDistance} are normalized fractions of the near-far depth range, not world units; \code{camera.near} was raised from $0.1$ to $5$ against \code{far}$=5000$ to stop depth precision from saturating; the kernel radius shrank from $90$ to $12$ to remove dark halos from surfaces reading as local occlusion behind a silhouette. \code{renderer.antialias} has no effect once rendering goes through the composer's targets, so \code{SMAAPass} runs last instead.

\subsection{Textures}

Two kinds: \textbf{downloaded} (ball color map, court diffuse map) and \textbf{procedurally generated} (ball normal/metalness/roughness; robot, bench and lamp-post normal/roughness/bump maps, built at runtime from a height field).

\subsection{Procedural Skybox}

The flat background color was replaced with the \code{Sky} addon (Preetham atmospheric scattering, no HDR download). The sun direction in the sky is always derived from the real \code{DirectionalLight} position, never tuned independently, so the sky's glow and the cast shadows stay aligned by construction. At Night the Preetham model degenerates (built for daylight scattering), so the sky mesh swaps to a flat dark color once the animated transition completes.

\section{Animation}
\label{sec:animation}
% =============================================================================

Every animation runs imperatively inside \code{requestAnimationFrame}; nothing is imported.

\begin{table}[htbp]
\centering
\small
\begin{tabularx}{\textwidth}{l X}
\toprule
\textbf{System} & \textbf{Technique} \\
\midrule
Automatic dribble & Push/drop/rise finite-state machine at fixed 120\,Hz, synced with the elbow/link1 motion \\
Shot flight & Projectile motion under constant gravity, sub-stepped at 240\,Hz, full collision response \\
Wheel steering & Exponential-smoothing angular interpolation, shortest path, wrap-safe \\
Legged gait & Diagonal-pair trot, advancing only while position actually changes frame to frame \\
Drone flight feel & Body bank and thrust-tilt composed via quaternion multiplication, not Euler angles, so the tilt axis stays attached to the nose at any yaw \\
Special moves & Dash (6$\times$-speed burst, 4\,s cooldown), Jump (parabolic arc, 5\,s cooldown), Flight (grab-rise-hold-descend, rigid ball attachment) \\
Shot animation & Windup/release/recover pose, elbow coupled to camera pitch while aiming \\
Time-of-Day fades & 2.5\,s smoothstep cross-fade of every light/sky parameter \\
Ball spin & Derived each frame from real displacement (axis $= \hat{u} \times \vec{v}$, rate $= |\vec{v}|/r$), consistent across every state with no per-state simulation \\
\bottomrule
\end{tabularx}
\caption{Animated systems; static geometry is limited to the court and lamp posts.}
\end{table}

\paragraph{Fixed timestep.} Integrating the dribble on the variable render \code{delta} produced a real bug: release velocity, a single-frame finite difference, collapsed to near zero whenever a frame hitch landed exactly on the push-easing saturation frame, and the ball dropped dead instead of bouncing. A $1/120$\,s accumulator (a slow frame runs several steps, a fast one may run none) removes the pathological case and makes the trajectory deterministic regardless of frame rate.

\paragraph{Third-person camera.} While dribbling, an orbit camera follows the robot (mouse yaw/pitch, \code{lookAt} pinned, pitch clamped to $\sim$3--51\textdegree). While aiming, it switches to a free orientation, yaw/pitch drive \code{camera.quaternion} directly, decoupled from position, allowing a look over the robot toward the hoop. Transitions always interpolate (position \code{lerp}, rotation \code{slerp}) toward a target recomputed every frame, since the two formulas are structurally different and switching without this layer snaps visibly.

\section{Physics, Collisions and Scoring}
\label{sec:physics}
% =============================================================================

No physics engine: every collision test is a closed-form formula, \textbf{sphere vs.\ AABB} (ball vs.\ backboard/walls/poles/benches, robot vs.\ static geometry) and \textbf{sphere vs.\ torus} (ball vs.\ rim), both using $\vec{v}' = \vec{v} - (1+e)(\vec{v}\cdot\hat{n})\hat{n}$ with a per-material restitution $e$ (backboard 0.15, rim 0.3, walls/poles 0.55, benches 0.5). The 66 wall/bleacher boxes are extracted mesh by mesh from the court GLTF, not one aggregate box, since the venue has intentional gaps an aggregate would misrepresent.

Shot flight is sub-stepped at 240\,Hz because at full speed a single frame can move the ball farther than the backboard is thick. Post-bounce collision cooldowns are tracked per object: a single global cooldown after a rim bounce would suspend the backboard check at the exact moment the deflected ball reaches it.

A hoop-assist potential field pulls an airborne shot toward the rim center, strength scaled by the shooter's \verb|shooting| stat. It is a clamped \emph{position} correction, not an acceleration: an acceleration accumulates with time spent in the cone, flinging slow close-range shots past the rim, while a pull clamped at the center cannot overshoot.

\textbf{Scoring}: 2 points if the shooter was inside the three-point arc at the moment of release, 3 otherwise, arc radius read from the court GLTF's accessor data. A basket is detected by interpolating the exact crossing point of the rim plane between two physics samples; testing only the sample already past the plane misses steep, close-range shots that genuinely went in.

\section{Audio}
\label{sec:audio}
% =============================================================================

Five sound effects, all synthesized at runtime with the Web Audio API, no audio file shipped: \textbf{score} (two ascending sine notes), \textbf{bounce} (low downward-sweeping tone plus a low-pass noise transient), \textbf{shoot} (band-pass noise sweeping upward), \textbf{steal} (same technique sweeping downward, shorter), \textbf{block} (low-pass noise hit over a descending tone). Each has a short linear attack before its exponential decay; an instant jump to peak gain produces an audible click. Background music is not yet implemented (Section~\ref{sec:conclusions}).

\section{1v1 Mode and Enemy AI}
\label{sec:ai}
% =============================================================================

The opponent runs a small tactical state machine, \code{EnemyState} $\in$ \{\code{CHASE\_BALL}, \code{ATTACK}, \code{DEFEND}\}, recomputed every frame from ball possession: chase a loose ball, attack toward its hoop and shoot in range, or position between the player and the player's hoop while attempting Steal or Block. Shot elevation is solved with the ballistic-projectile equation at fixed speed, falling back to a 50\textdegree{} arc if the target is unreachable.

Steal and Block share one module between player and AI: a reach/sweep animation, a bounding-box contact test (asymmetric for Steal, wide margin in front of the sweeping robot, almost none behind), and a resolve step re-validating possession at the end of the sweep rather than at first contact. Robot-vs-robot overlap resolves with a possession-based asymmetric correction (the ball holder yields 25\% of the separation, the other robot 75\%); a robot mid-shot is never displaced, since its position also drives the chase camera.

The two choices that shaped the project most: building everything procedurally (robots, textures, skybox, sounds; only the court and ball are external), and simulating on fixed timesteps decoupled from the render rate. A recurring lesson, met independently in the shadow setup, SSAO configuration and light intensities, is that a graphics library's physically-motivated defaults assume a scene scale; importing an asset with different units means re-deriving each such parameter rather than tuning it blindly.

Known limitations: no 3v3 mode; no background music, only the synthesized SFX of Section~\ref{sec:audio}; robot color customization does not persist across a page refresh, in-memory for the session only.
